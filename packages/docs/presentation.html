Welcome. Boxes is a graph editor built on top of Cytoscape.js. Today I'll walk through what it does, show it working live in these slides, and cover the main use cases.
The Problem

Graphs are everywhere.

Knowledge graphs · Ontologies · Dependency trees · Org charts · API topologies

  • 😩 Building a graph editor from scratch takes weeks
  • 🎨 Cytoscape.js is powerful — but its stylesheet is separate from your data
  • 🔒 Styles get lost when you export or share graph data
  • 🔌 No standard format other tools can produce and exchange
Graphs appear everywhere in modern software but building editors for them is hard. Cytoscape.js is the go-to library but it leaves styling as a separate concern, disconnected from the data.
The Solution

Boxes = Cytoscape.js
+ CSS as data

  • ✅ Full editing UI out of the box
  • 🎨 Stylesheet lives inside the graph file
  • 💾 One JSON format — .boxes
  • 🔗 Embed anywhere in 3 lines of HTML
  • 🧩 Vue 3 & React 18 components included
  • 🌐 Export to RDF, JSON-LD, Arrows, PDF
myGraph.boxes
{
  "elements": { nodes, edges },
  "userStylesheet": [ rules… ],
  "lastLayout": { name, options },
  "context": { prefixes… }
}
Boxes packages CSS styling rules into the graph data model itself — the stylesheet travels with the graph. There's one JSON format, the .boxes file, and it maps directly onto Cytoscape.js options.

The .boxes Format

That graph is just JSON

{
  "elements": {
    "nodes": [
      { "data": { "id": "js",   "label": "JavaScript", "type": "lang" }, "position": { "x": 300, "y": 200 } },
      { "data": { "id": "ts",   "label": "TypeScript",  "type": "lang" }, "position": { "x": 160, "y": 100 } },
      { "data": { "id": "node", "label": "Node.js",     "type": "rt"   }, "position": { "x": 160, "y": 300 } },
      { "data": { "id": "react","label": "React",       "type": "lib"  }, "position": { "x": 440, "y": 100 } },
      { "data": { "id": "vue",  "label": "Vue",         "type": "lib"  }, "position": { "x": 440, "y": 300 } }
    ],
    "edges": [
      { "data": { "id": "e1", "source": "ts",   "target": "js",   "label": "compiles to" } },
      { "data": { "id": "e2", "source": "node", "target": "js",   "label": "built on"    } },
      { "data": { "id": "e3", "source": "react","target": "js",   "label": "built on"    } },
      { "data": { "id": "e4", "source": "vue",  "target": "js",   "label": "built on"    } }
    ]
  },
  "userStylesheet": [
    { "selector": "node[type='lang']", "style": { "background-color": "#4a7fd4", "shape": "ellipse" } },
    { "selector": "node[type='rt']",   "style": { "background-color": "#34d399", "shape": "rectangle" } },
    { "selector": "node[type='lib']",  "style": { "background-color": "#fb923c", "shape": "roundrectangle" } }
  ],
  "lastLayout": { "name": "preset" }
}
The graph you just saw is described entirely in this JSON. The nodes, edges, styles and layout are all self-contained in a .boxes file.

Use Case 2

Embed in 3 lines

One CDN script. One container div. One fetch call.

<!-- 1. CDN script — the only dependency -->
<script src="https://unpkg.com/cytoscape@3/dist/cytoscape.min.js"></script>

<!-- 2. Container -->
<div id="graph" style="width:100%;height:500px"></div>

<!-- 3. Load and render -->
<script>
  fetch('my-graph.boxes')
    .then(r => r.json())
    .then(({ elements, userStylesheet, lastLayout }) => {
      cytoscape({
        container: document.getElementById('graph'),
        elements,
        style:  userStylesheet,
        layout: lastLayout ?? { name: 'preset' }
      });
    });
</script>
Because userStylesheet maps directly to Cytoscape's style option, embedding is literally three things: the CDN, a div, and a fetch. No framework, no build tools.

… like this

fetch('my-graph.boxes').then(r=>r.json()).then(({elements,userStylesheet,lastLayout})=>{

Model OWL / RDFS / SKOS ontologies visually — then export to standards-compliant RDF

OWL mini-ontology · live render
  • 🦉 Pre-styled types for owl:Class, owl:ObjectProperty, owl:DatatypeProperty
  • 🔗 rdfs:subClassOf, rdfs:domain, rdfs:range edge types
  • 🌐 JSON-LD context maps prefixes to URIs
  • 📤 Export as RDF/Turtle, JSON-LD, or RDF/XML
  • 📥 Import existing Turtle or JSON-LD files
  • 📎 Round-trip: edit → export → import → edit
The OWL Ontology template ships pre-configured with all the node and edge types an ontologist needs, plus JSON-LD context for common prefixes.

Developer Integration

Vue 3 & React 18

Vue 3

<template>
  <BoxesEditor
    :elements="graph.elements"
    :node-types="nodeTypes"
    :edge-types="edgeTypes"
    @change="onGraphChange"
    ref="editorRef"
  />
</template>

<script setup>
import { BoxesEditor } from 'boxes-vue';

async function exportPdf() {
  const blob = await editorRef.value
    .editor.exportPdf({ paperSize: 'A4' });
  // trigger download...
}
</script>

React 18

import { BoxesEditor } from 'boxes-react';
import { useRef } from 'react';

export function GraphPanel({ graph }) {
  const editorRef = useRef(null);

  return (
    <BoxesEditor
      ref={editorRef}
      elements={graph.elements}
      nodeTypes={nodeTypes}
      edgeTypes={edgeTypes}
      onChange={handleChange}
    />
  );
}
Both wrappers expose the full BoxesEditor instance imperatively via ref.

First-class Vue 3 and React 18 components wrap the core library. Both use refs to expose the imperative BoxesEditor API for operations like exportPdf and importGraph.
